# 机械臂动作自动规划智能体的设计与实现

## 摘要

本课程作业围绕“机械臂动作自动规划”实现了一个面向乐白 LM3 机械臂的 Web 智能体原型。系统使用 Three.js 构建机械臂数字孪生和桌面抓取场景，使用结构化 Scene Graph 描述场景中物体的位置、颜色、类别和尺寸，再调用千问模型完成自然语言任务理解。为了避免大模型直接输出不可验证的底层控制量，系统采用“LLM 任务规划 + 工具层校验 + 运动学求解 + Web 仿真执行”的分层设计：LLM 负责理解任务和生成任务级步骤，工具模块负责将步骤转换为笛卡尔路点和 LM3 六关节轨迹，并对规划结果进行安全校验。

项目还加入了一个手外眼虚拟相机。该相机以桌面外侧支架的形式固定在工作台上方，可以对当前 3D 场景进行截图，并作为可选视觉提示发送给支持图像输入的千问多模态模型。课程版仍以结构化 Scene Graph 作为主要可靠输入，但该功能模拟了真实机器人系统中“RGB-D 图像输入 VLM，再结合结构化空间信息进行任务规划”的流程。实验结果表明，系统能够完成“把红色方块放到蓝色方块上”等典型抓取放置任务，并输出规划 JSON、轨迹评分、校验结果和执行日志。

关键词：智能体；机械臂；任务规划；Three.js；多模态大模型；数字孪生

## 1. 什么是智能体

本课程中所说的智能体，不只是一个能聊天的大模型，而是一个能够围绕目标任务主动调用工具、读取知识、执行流程并反馈结果的软件系统。按照本项目的理解，一个专业智能体至少包含四个部分：LLM、工具、知识库和工作流。

LLM 负责理解用户意图。例如用户输入“把红色方块放到蓝色方块上”，大模型需要识别出目标物体、目标位置和任务类型。工具负责把模型的语言理解变成可执行操作，例如轨迹生成、逆运动学求解、可达性校验和日志记录。知识库提供领域约束，例如机械臂工作半径、桌面高度、夹爪开合逻辑、物体尺寸和安全高度。工作流则规定这些模块如何按顺序配合，避免模型直接越过校验步骤控制机械臂。

因此，智能体的关键不是“让模型直接做所有事”，而是让模型在合适的位置做决策，再把具体计算和执行交给更可靠的工具模块。本项目的机械臂动作规划智能体正是按照这个思路设计的：LLM 负责上层任务规划，工具层负责运动学和仿真执行，Web 页面负责展示状态、日志和结果。

## 2. 自己搭建的专业智能体

本项目搭建的是一个面向机械臂抓取放置任务的专业智能体，应用场景不是日常办公或娱乐，而是机械臂任务规划。系统的输入包括自然语言任务、结构化 3D 场景图，以及可选的手外眼相机截图。系统的输出包括任务步骤、末端轨迹、关节轨迹、校验结果和仿真执行日志。

系统采用纯前端技术实现，主要技术栈包括 TypeScript、Vite、Three.js、Cannon-es 和 Vitest。选择纯前端实现的原因是课程作业需要方便展示和部署，GitHub Pages 可以直接托管静态网页。DashScope API Key 由用户在页面中输入并保存在浏览器本地存储中，不需要后端服务器保存密钥。这样的方案降低了部署成本，也方便课堂演示。

系统总体流程如下：

```mermaid
flowchart LR
  A["用户自然语言任务"] --> B["Scene Graph"]
  C["手外眼相机截图"] --> D["LLM/VLM 任务规划"]
  B --> D
  D --> E["语义步骤 steps"]
  E --> F["笛卡尔路点 cartesianWaypoints"]
  F --> G["LM3 数值 IK"]
  G --> H["jointTrajectory"]
  H --> I["Web 仿真执行"]
  I --> J["日志与 JSON 输出"]
```

其中 Scene Graph 是系统最稳定的感知输入，包含每个物体的 ID、颜色、类型、位置、尺寸和可移动状态。手外眼相机截图作为额外视觉提示，用于展示多模态模型输入流程，而不是替代结构化数据。

## 3. 功能描述和技术分析

### 3.1 结构化场景图

在 3D 场景中，每个物体都对应一个结构化对象。例如红色方块包含 `id=red_cube`、`type=cube`、`color=red`、`position` 和 `size` 等字段。用户可以拖动物体，也可以添加新的方块或圆柱。右侧场景表会实时显示物体状态。

这个设计模拟了真实机器人系统中感知模块的输出。在真实环境中，RGB-D 相机和视觉检测模型可以识别物体并估计位置；在课程版中，我用手动维护的 Scene Graph 代替真实感知结果。这样做的好处是结果可控，便于校验和演示，也能把重点放在智能体工作流上。

### 3.2 LLM 与多模态视觉提示

系统通过 DashScope 的 OpenAI-compatible 接口调用千问模型。提示词要求模型只输出 JSON，避免生成无法执行的长段自然语言。模型输出的每一步包含动作名、物体 ID、目标位置、夹爪动作和中文描述。

为了模拟真实 VLM 输入，项目增加了一个手外眼虚拟相机。相机固定在桌面外侧的支架上，包括底座、立杆、横向支臂和相机头，朝向工作区拍摄。用户点击拍照后，Three.js 会从该相机视角离屏渲染当前场景，生成 JPEG 图片。如果勾选“发送视觉提示”，系统会把图片作为 `image_url` 与 Scene Graph 一起发送给支持图像输入的千问模型。

这里的设计重点不是让图片代替结构化坐标，而是展示一种更接近真实具身智能系统的数据形式：图像提供直观场景上下文，Scene Graph 提供可校验的空间信息。

### 3.3 工具化规划与校验

系统不会直接执行 LLM 输出的文本，而是把规划结果转换为三层数据：

- `steps`：面向用户解释的语义步骤。
- `cartesianWaypoints`：末端执行器的笛卡尔路点，包含夹爪开合状态。
- `jointTrajectory`：通过 LM3 数值逆运动学生成的六关节角序列。

校验模块会检查规划是否包含完整的抓取和释放动作、引用的物体是否存在、轨迹点是否低于桌面、目标是否超过工作半径，以及 IK 是否可达。如果校验失败，系统不会继续执行，而是在日志中输出修复提示。

### 3.4 LM3 数字孪生与运动执行

项目使用课题组已有的乐白 LM3 GLB 模型作为机械臂外观，并映射 `Joint1` 到 `Joint6` 的关节节点。为了让轨迹和机械臂动画更一致，系统从 GLB 中提取了关节层级、关节轴和近似 TCP 节点。执行时，系统按 `jointTrajectory` 插值驱动关节。夹爪闭合后，目标物体会跟随 TCP 移动；释放时，系统按几何关系把物体稳定放在目标物体上方。这里没有追求完整物理仿真，因为轻量物理引擎在方块堆叠时容易出现滑落。课程版优先保证规划流程展示稳定。

### 3.5 日志与可解释性

系统执行过程中会输出输入 Scene Graph、手外眼相机快照元数据、规划响应 JSON、轨迹评分、校验结果和每一步动作日志。这些日志不仅用于调试，也让智能体的决策过程更容易解释。相比只展示最终动画，日志可以说明系统为什么这样规划，以及每一步是否通过工具校验。

## 4. 测试与结果

项目使用 Vitest 对核心模块进行了测试，包括 API Key 管理、多模态请求构造、中文指令解析、本地规划、轨迹生成、LM3 运动学和 JSON 日志格式化。验证命令如下：

```powershell
cd web
npm.cmd test
npm.cmd run build
```

当前测试结果为 7 个测试文件、20 个用例通过；构建结果也通过。实际运行中，用户可以完成以下流程：

1. 在 3D 场景中拖动物体或添加物体。
2. 点击“拍照”，从手外眼相机视角生成场景截图。
3. 输入“把红色方块放到蓝色方块上”。
4. 系统输出规划 JSON、轨迹评分和执行日志。
5. Web 场景中机械臂按照关节轨迹完成抓取和放置。

## 5. 开发过程中遇到的问题

第一个问题是模型选择。项目早期曾考虑使用通用机械臂资源，但这和实验室真实设备关联不强。后续改为乐白 LM3 模型，使课程项目和后续毕业设计方向更一致。

第二个问题是动画和轨迹不匹配。仅仅画出末端轨迹并不能说明机械臂真的按轨迹运动。为了解决这个问题，项目加入 LM3 正逆运动学，把笛卡尔路点转换成关节角，再由关节角驱动模型。

第三个问题是物体堆叠。方块放到另一个方块上时，物理引擎可能因为摩擦和碰撞误差导致滑落。最终我采用静态几何放置，保证课程演示稳定，并在论文中明确说明这不是完整物理仿真。

第四个问题是如何体现多模态输入。如果直接依赖真实 RGB-D 相机，课程作业难度会明显增加。因此我先在 Web 场景中实现手外眼虚拟相机，用它模拟真实视觉输入流程，为后续真实相机接入保留思路。

## 6. 讨论和心得

这次作业让我认识到，使用 LLM 做专业工作流自动化，关键是“快、好、省”之间的平衡。

“快”体现在开发过程中可以让 LLM 辅助分析方案、生成代码骨架、定位 bug 和整理文档。例如项目中从通用机械臂方案转向乐白 LM3 方案时，LLM 帮助我比较了课程版和毕业设计版的不同边界，避免一开始就把真实机器人控制做得过重。

“好”体现在不能完全相信模型输出，而要用工具校验结果。本项目中，LLM 只负责生成任务级规划，真正的可达性、轨迹点、关节角和执行过程都由确定性工具模块处理。这样可以减少模型幻觉对系统执行的影响。

“省”体现在用合适的简化方案完成课程目标。比如课程版没有直接接入真实机械臂，也没有做完整 RGB-D 感知，而是用 Scene Graph 和虚拟手外眼相机模拟感知输入。这样既能展示智能体完整流程，又不会把时间消耗在硬件标定、驱动安全和物理仿真细节上。

我也意识到，课程论文不能只写“系统用了大模型”，而要说明大模型在工作流中的具体位置。一个专业智能体的价值不在于模型回答得多漂亮，而在于它能否把 LLM、工具、知识和流程组织成可验证的自动化系统。

## 7. 展望

如果后续把这个课程项目扩展为毕业设计，可以从以下方向继续推进：

1. 将虚拟手外眼相机替换为真实 RGB-D 相机，完成相机内参标定和手眼标定。
2. 使用视觉检测或分割模型从 RGB-D 图像中自动生成 Scene Graph，而不是手动维护物体位置。
3. 将 Web 中的 `cartesianWaypoints` 和 `jointTrajectory` 发送到 Python/FastAPI 后端，再由后端调用乐白 `lebai_sdk`。
4. 在真实执行前加入 dry-run、安全区域、速度限制、急停和人工确认机制。
5. 进一步研究类似 RoboBrain 的多模块设计，将任务规划、可供性感知和轨迹预测拆分为可训练或可替换的专业模块。

目前课程版的价值在于完成了一个可运行、可解释的最小闭环：自然语言输入、结构化场景理解、多模态视觉提示、任务规划、运动学转换、仿真执行和日志记录。它还不能等同于真实机器人系统，但为后续 sim-to-real 留出了比较清晰的接口。
