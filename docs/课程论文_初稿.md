## 摘要

本课程作业围绕“机械臂动作自动规划”实现了一个面向乐白 LM3 机械臂的 Web 智能体原型。系统使用 Three.js 构建机械臂数字孪生和桌面抓取场景，用结构化 Scene Graph 描述场景中物体的位置、颜色、类别、尺寸和当前机械臂状态，再调用千问模型完成自然语言任务理解。为了避免大模型直接输出不可验证的底层控制量，系统采用“LLM 任务理解 + 工具层轨迹生成 + 运动学求解 + Web 仿真执行”的分层结构：LLM 负责把用户指令转化为任务级意图，工具模块负责生成笛卡尔路点、关节轨迹和校验结果，前端仿真负责展示执行过程。

项目以实验室实际购置的乐白 LM3 机械臂和夹爪为后续落地方向，当前课程版不直接控制真实设备，而是在 Web 端完成可解释、可运行的最小闭环。系统支持“把红色方块放到蓝色方块上”“夹起红色方块”“把当前夹持物体往右放一点”等任务，并输出包含 `start_tcp`、`cartesianWaypoints`、`jointTrajectory`、评分和校验信息的规划 JSON。项目还实现了一个固定在桌面外侧支架上的手外眼虚拟相机，可从 3D 场景中生成截图，并作为可选视觉提示发送给支持图像输入的千问多模态模型。该功能用于模拟真实机器人系统中“视觉感知 + 结构化空间信息 + 任务规划”的数据流。

关键词：智能体；机械臂；任务规划；Scene Graph；Three.js；千问；数字孪生

## 1. 什么是智能体

我理解的智能体并不是简单的聊天机器人，而是一个围绕目标任务组织模型、工具、知识和流程的软件系统。它需要能理解用户意图，也要能调用外部工具，把模型输出变成可检查、可执行、可反馈的结果。对机械臂任务来说，如果只让大模型回答“先抓取，再放置”，系统并没有真正完成动作规划；只有当它能把任务拆成明确步骤、生成轨迹、检查可达性，并在仿真或真实设备上执行时，才更接近课程要求中的专业智能体。

本项目中的智能体由四个部分组成。

第一是 LLM。它负责理解自然语言任务，例如从“把红色方块放到蓝色方块上”中识别目标物体、放置目标和任务类型。第二是工具。工具包括 Scene Graph 解析、抓取点生成、笛卡尔路点生成、LM3 数值逆运动学、轨迹评分和安全校验。第三是知识。知识不是一个单独的文档库，而是写入系统约束中的领域知识，例如 LM3 工作半径、桌面高度、安全抬升高度、夹爪开口限制、方块堆叠几何关系和后续乐白 SDK 接入边界。第四是工作流。系统按“输入场景与任务、生成计划、校验计划、必要时修复、执行仿真、记录日志”的顺序运行，避免模型输出直接越过工具层进入执行层。

这种设计的关键是把大模型放在合适的位置。LLM 擅长处理语言和任务意图，但不适合直接承担全部几何计算和运动控制。因此本项目没有让模型直接输出机械臂每个关节角，而是让它做上层规划，再由确定性的工具模块完成可达性和轨迹计算。这样做的好处是系统出了问题时可以定位：是任务理解错误、场景数据错误、IK 不可达，还是执行动画问题。

## 2. 自己搭建的专业智能体

### 2.1 系统定位

本项目搭建的是一个面向机械臂抓取放置任务的专业智能体，不是娱乐产品，也不是日常办公助手。课程版目标是展示机械臂动作规划的完整工作流，而不是一次性完成真实机器人落地。系统运行在浏览器中，主要技术栈为 TypeScript、Vite、Three.js、Cannon-es 和 Vitest。选择纯前端方案的原因是课程作业需要方便展示和部署，GitHub Pages 可以直接托管静态页面，用户也可以在页面中输入自己的 DashScope API Key，不需要把密钥放到服务器。

当前系统包含以下功能：

- 乐白 LM3 机械臂 GLB 模型加载和六关节动画。
- 通用平行夹爪开合动画与顶部抓取逻辑。
- 可拖拽的 3D 桌面场景，支持方块和圆柱。
- Scene Graph 实时记录物体、机械臂 TCP、当前关节角和夹持状态。
- 自然语言任务输入，优先调用千问模型，失败时回退到本地规则规划。
- Plan / Validate / Repair / Execute 工作流展示。
- 规划 JSON、输入 Scene Graph JSON、校验结果和执行日志输出。
- 固定手外眼虚拟相机，可生成场景截图并作为多模态视觉提示。

### 2.2 系统工作流

系统的核心流程可以概括为：

```mermaid
flowchart LR
  A["用户自然语言任务"] --> B["Scene Graph"]
  C["手外眼相机截图"] --> D["LLM/VLM 任务理解"]
  B --> D
  D --> E["任务级步骤 steps"]
  E --> F["笛卡尔路点 cartesianWaypoints"]
  F --> G["LM3 数值 IK"]
  G --> H["关节轨迹 jointTrajectory"]
  H --> I["Web 数字孪生执行"]
  I --> J["日志与 JSON 输出"]
```

这里 Scene Graph 是系统最稳定的感知输入。它包含物体 ID、颜色、类型、位置、尺寸、是否可移动，以及机械臂当前 TCP 和关节角。虚拟相机截图是额外视觉提示，用来模拟真实 VLM 输入，而不是取代结构化坐标。这样做比较符合课程作业的难度：既能展示多模态智能体的形式，又不会把主要精力消耗在真实 RGB-D 检测和相机标定上。

### 2.3 规划 JSON 设计

系统输出的规划结果不是一段自然语言，而是结构化 JSON。主要字段包括：

- `steps`：面向用户解释的任务步骤，例如移动到目标上方、下降、抓取、抬升、移动到放置点、释放。
- `trajectory`：用于前端画线的末端路径。现在该路径从 `start_tcp` 开始，而不是从目标物体附近直接开始。
- `cartesianWaypoints`：更接近执行层的数据，每个路点包含 ID、位置、夹爪状态、速度、融合半径和坐标语义。
- `jointTrajectory`：由 LM3 数值逆运动学求解得到的六关节轨迹点。
- `score`：轨迹评分，包括可达性、平滑度、碰撞和关节限制状态。
- `validation`：校验结果，说明规划是否可执行。

我在实现过程中发现，最初轨迹直接从抓取物体附近开始，这在演示上会造成误解：真实机械臂不可能凭空出现在物体上方，它应该从当前夹爪位置出发。因此终版中加入了 `start_tcp`。这个点来自当前数字孪生的 TCP 和关节角，规划 JSON 也会保留该起点。这样末端轨迹和物体轨迹被区分开来：末端轨迹从当前夹爪开始，物体轨迹只在抓取成功后才跟随夹爪移动。这个调整虽然不复杂，但对后续 sim-to-real 很重要，因为真实机械臂执行轨迹时必须知道当前状态。

### 2.4 LLM 与视觉提示

系统通过 DashScope 的 OpenAI-compatible 接口调用千问模型。提示词要求模型尽量输出 JSON，而不是输出大段解释性文字。课程版默认模型可以由用户在界面中填写，例如文本模型用于任务理解，多模态模型可用于接收视觉提示。

手外眼相机固定在桌面外侧的支架上，朝向整个工作区。用户点击“拍照”后，Three.js 会从该相机视角离屏渲染当前 3D 场景，生成 JPEG 图片。如果勾选“发送视觉提示”，系统会把图片以 `image_url` 的形式与 Scene Graph 一起发送给模型。这个设计的意义在于展示真实机器人系统中可能出现的数据流：真实环境里可以由 RGB-D 相机提供图像和深度，再由视觉检测模块生成结构化物体位置；课程版先用虚拟相机和 Scene Graph 模拟这一流程。

需要说明的是，当前系统不依赖图片来计算物体坐标。物体位置仍以 Scene Graph 为准，因为它更稳定、可校验，也便于课程演示。如果完全依赖截图让模型猜三维坐标，结果会不可靠，也很难解释。

### 2.5 运动学和仿真执行

机械臂模型使用课题组已有的乐白 LM3 GLB 资源，程序映射了 `Joint1` 到 `Joint6` 的关节节点，并根据关节层级和近似 TCP 位置实现数值正逆运动学。规划器先生成笛卡尔路点，再通过 IK 转换为关节角序列。执行时，Three.js 模型不再只是让夹爪或物体瞬移，而是按关节角插值运动，物体在抓取后绑定到夹爪中心附近。

当前夹爪没有做完整物理抓取。真实夹爪抓取涉及接触、摩擦、夹紧力和物体质量，Web 端轻量物理引擎很难稳定表现这些细节。课程版采用几何绑定方式：夹爪闭合并通过抓取姿态校验后，目标物体跟随夹爪移动；释放时按几何关系放置到目标表面。这个取舍牺牲了一部分物理真实性，但保证了演示稳定，也让重点回到智能体工作流本身。

## 3. 功能描述和技术分析

### 3.1 本地规则规划与 LLM 规划的关系

系统优先使用 LLM 进行任务理解。如果模型不可用、API Key 未填写，或者模型不支持当前输入，系统会回退到本地规则规划。本地规则规划不是为了取代 LLM，而是为了保证课程演示稳定。它支持常见任务类型：

- `pick_place`：把一个物体放到另一个物体上。
- `pick_only`：夹起某个物体并保持夹持。
- `relative_place`：把当前夹持物体或指定物体往右、左、前、后移动一点。

这种设计也便于调试。LLM 出错时，系统仍能通过本地规则完成演示；本地规则生成的 JSON 可以作为期望格式，帮助检查 LLM 输出是否合规。

### 3.2 校验与修复机制

系统不会无条件执行规划结果。校验模块会检查以下问题：

- 目标物体是否存在。
- 被操作物体是否可移动。
- 任务是否包含必要动作，例如抓取任务需要 `grasp`，放置任务需要 `release`。
- 轨迹点是否低于桌面安全高度。
- 目标点是否超过机械臂工作半径。
- 逆运动学是否出现严重不可达误差。

如果校验失败，页面会停在 Repair 阶段并输出原因。这个机制体现了工具层的重要性：大模型可以给出想法，但执行前必须经过确定性模块过滤。对机械臂系统而言，这比让模型直接控制执行更安全。

### 3.3 可解释日志

项目右侧执行日志会记录用户任务、输入 Scene Graph JSON、规划响应 JSON、校验结果和每一步动作。这样设计的原因是，课程论文和答辩不能只展示一个动画，而要说明系统如何从语言输入变成可执行轨迹。例如，当系统输出 `start_tcp -> pre_grasp -> grasp -> lift -> pre_place -> place -> retreat` 时，可以清楚解释末端执行器从哪里出发、何时抓取、何时释放。日志也方便定位问题，例如 IK 不可达、夹爪中心偏差、物体堆叠穿模等。

### 3.4 测试结果

项目使用 Vitest 测试核心模块，包括 LLM 请求构造、Scene Graph 解析、抓取规格、TCP 校准、LM3 运动学、规划器和 UI 日志格式。当前验证命令如下：

```powershell
cd web
npm.cmd test
npm.cmd run build
```

最近一次运行结果为 9 个测试文件、34 个测试用例全部通过，生产构建也通过。Vite 会提示打包后的 JavaScript chunk 超过 500 kB，这是 Three.js 和 3D 资源相关的体积提示，不影响课程展示。后续如果要优化加载速度，可以再做动态导入或资源拆分。

## 4. 开发过程中遇到的问题

第一个问题是模型选择。项目早期考虑过使用通用机械臂资源，但这和实验室实际设备关系不强。后来改为围绕乐白 LM3 和夹爪设计，这样课程作业和后续毕业设计方向更一致。虽然课程版仍是 Web 仿真，但数据结构已经为后续乐白 Python SDK 接入留出接口。

第二个问题是轨迹和动画不一致。最初系统可以画出末端路径，但机械臂关节动画和路径不完全匹配，看起来像物体被吸附或瞬移。后来加入 LM3 正逆运动学和关节轨迹，让模型按关节角运动。同时，执行阶段避免每一帧重新选择不同 IK 分支，减少了机械臂姿态闪烁。

第三个问题是夹爪姿态。完整 6D 姿态 IK 难度较高，需要精确的 DH 参数、TCP 标定、关节零位和姿态约束。课程版目前采用顶部抓取为主，尽量让夹爪朝下，并通过 TCP 校准参数调整夹爪中心。这个方案不是工业级控制，但能支撑课程演示。

第四个问题是物体堆叠。方块放到另一个方块上时，如果完全依赖轻量物理引擎，可能因为摩擦和碰撞误差滑落。最终采用静态几何放置，保证演示稳定。论文中必须说明这一点，不能把它说成完整物理仿真。

第五个问题是如何体现 LLM 的作用。如果只是本地规则规划，项目会像一个普通仿真工具；如果完全依赖 LLM，系统又不稳定。最终方案是让 LLM 位于任务理解层，本地工具负责轨迹和校验。这样既能体现智能体属性，也保留了工程可靠性。

## 5. 讨论和心得

这次作业让我比较清楚地认识到，LLM 在专业工作流中的价值不是“替人写一段看起来很完整的答案”，而是参与一个有约束、有反馈的自动化流程。机械臂规划尤其明显，因为错误的输出不只是文字错误，还可能变成不可达轨迹、碰撞风险或错误抓取。

在开发中，我把“快、好、省”理解为三个不同层面的取舍。

“快”体现在用 LLM 辅助方案分析、问题定位和文档整理。例如从通用机械臂方案转向乐白 LM3 方案时，LLM 帮助我比较了课程版和毕业设计版的边界，避免一开始就把真实机器人控制、相机标定和后端框架都做进去。

“好”体现在不盲目信任模型输出。系统要求规划结果必须经过工具校验，轨迹和关节角也由程序计算。大模型负责理解“做什么”，工具负责判断“能不能做、怎么做”。这种分工比让模型直接生成控制指令更合理。

“省”体现在选择合适的简化方案。课程版没有做真实 RGB-D 感知，也没有完整物理夹爪，而是用 Scene Graph、虚拟相机和几何抓取完成闭环。这样可以把有限时间集中在智能体工作流、可解释日志和可部署演示上。

我也意识到课程论文不能写成空泛的“系统使用了大模型，提升了效率”。更重要的是写清楚大模型在哪里起作用，哪些部分由工具完成，哪些地方做了工程取舍，当前系统还有哪些限制。这个项目目前还不是一个真实工业机器人系统，但它已经把自然语言输入、场景理解、任务规划、轨迹生成、校验、仿真执行和日志解释串成了一个可运行流程。

## 6. 展望

如果后续把这个方向扩展为毕业设计，我认为可以从以下几个方面继续推进。

第一，接入真实 RGB-D 相机。当前虚拟手外眼相机只是模拟视觉输入，后续可以使用真实相机完成内参标定和手眼标定，再由检测或分割模型生成 Scene Graph。

第二，接入乐白 Python SDK。当前 JSON 中已经保留 `cartesianWaypoints` 和 `jointTrajectory`，后续可以通过 FastAPI 后端接收规划结果，再调用 `lebai_sdk` 或 `lebai_sdk_asyncio` 执行真实机器人动作。真实执行前必须增加 dry-run、安全区域、速度限制、急停和人工确认。

第三，改进 IK 和姿态约束。课程版的数值 IK 能支撑演示，但还不是完整 6D 姿态控制。后续应结合乐白 LM3 的真实关节参数、TCP 标定和夹爪安装偏移，提高夹爪姿态精度。

第四，研究更细的任务规划能力。类似 RoboBrain 的工作把任务规划、可供性感知和轨迹预测拆成不同模块。本项目可以在课程版基础上继续扩展，例如让系统自动选择抓取面、估计更合理的抓取点，或者把失败日志反馈给 LLM 生成修复计划。

总体来说，本项目完成了一个课程级但可运行的机械臂规划智能体。它没有夸大为真实机器人系统，而是把当前能力和限制区分清楚：课程版完成 Web 数字孪生和任务规划闭环，毕业设计版可以沿着真实感知、后端控制和 sim-to-real 的方向继续落地。
